## 一、全局认知

### 1.1 项目本质

**一句话定义**：Y.Mine 是一套以金融精算为内核、以元架构迁移为方法论、以纯静态HTML为载体的**商业决策操作系统**。

它不是一个传统意义上的"网站"或"作品集"，而是作者将跨财务、MBA、神经科学、博弈论、计算机的认知体系，物化成的一个**可交互、可计算、可验证的决策外骨骼**。

### 1.2 核心底层逻辑（六层架构）

| 层级 | 核心模块 | 关键概念 |
|------|---------|---------|
| L0 神经科学 | [circle-lab.html](../labs/evidence/circle-lab.html) | 小脑-皮层运动-认知耦合、行为代理模型 |
| L1 信号提纯 | [info-funnel.html](../labs/evidence/info-funnel.html) | 7层信息漏斗，100%噪声→20%纯信号 |
| L2 精算内核 | [marketing-reinvented.html](../labs/evidence/marketing-reinvented.html) | 6维风险向量 + 圆锥浓度模型 + CAPM |
| L3 铁律阈值 | 算法层内置 | 0.48生死线 / 0.50定价权基准线 / 0.68稳态上限线 |
| L4 案例对标 | [case-library.html](../labs/evidence/case-library.html) | 16案例6维向量矩阵，余弦相似度+欧氏距离匹配 |
| L5 输出闭环 | 定价模拟器 | 5输入→4输出，6灯风控熔断 |

### 1.3 核心算法清单

| 算法 | 公式 | 实现位置 |
|------|------|---------|
| 凯利公式 | f* = (bp - q) / b | GameMind、金融估值模拟器 |
| CAPM资本资产定价 | R = Rf + Rp × β | 金融估值模拟器 |
| 圆锥浓度模型 | C = m × ρ₀ / V(h) | 金融估值模拟器（视觉层） |
| 余弦相似度 | sim = cos(θ) × e^(-‖d‖) | 案例向量库 |
| 定价权乘数 | Multiplier = 1 + (C - 0.48) × 2.5 | 定价模拟器 |
| 三流分发框架 | 信息+流量+物理 = 全链路分发成本 | 瑞幸反向拆解器 |

### 2.3 结构性问题（按严重程度排序）

#### 🔴 P0：极端代码重复——侧边栏复制13次

13个HTML文件，每个文件内联约200行侧边栏HTML+CSS，13份完全相同。改一个导航项需要改13个文件。这是架构级反模式，不是代码规范问题。

**涉及文件**：所有13个HTML

#### 🔴 P0：单文件巨兽——marketing-reinvented.html超过1500行

该文件内聚了：
- 圆锥浓度Canvas可视化
- CAPM资本资产定价计算器
- 6维风险向量计算引擎
- 瑞幸反向拆解器（6层瀑布）
- 定价模拟器（5输入4输出）
- 风控熔断系统（6灯判定）
- localStorage总线逻辑
- 全部CSS + 全部侧边栏

这是**原型演化超过架构承载力**的典型症状。

#### 🟡 P1：无分层——算法与DOM揉在一起

没有前后端分离（因为根本没有后端），但也没有最基本的关注点分离：
- 数学计算、Canvas绘制、DOM操作、事件监听、localStorage读写全部耦合
- 例如[general-game-os.html](../labs/evidence/general-game-os.html)中5个Canvas模块，每个都是200行左右自包含IIFE，内部既有`ctx.fillStyle`又有凯利公式计算
- 7个总线对象全部挂在`window`全局，无命名空间、无模块系统、无状态管理

#### 🟡 P1：CSS重复度40%+

`.sidebar`、`#glow`、`.container`、`@keyframes fadeInUp`、`@keyframes floatBg`等基础样式在每个HTML中完整复制。估算全项目40-50%的CSS是冗余代码。

#### 🟡 P1：文档与代码不同步

- README标注v1.3，各HTML版本号不一致
- [marketing-reinvented.md](./marketing-reinvented.md)描述设计思路，HTML已演化出更多功能
- [pricing-decoder.md](./pricing-decoder.md)提到部分构想未在HTML中完全实现

#### 🟢 P2：模块边界概念驱动而非功能驱动

- value-pyramid.html既是"证据链"又像"知识总览"
- cross-domain-mapping.html既是验证又包含方法论
- binary-solver.html说是工具但只实现了鸡兔同笼

## 三、逻辑推演：数据流审计

### 3.1 设计中的理想数据流

```
🔭 信息漏斗（7层筛选）
    ↓ funnel-bus-update
🧪 金融估值模拟器（CAPM + 6维向量 + 圆锥浓度）
    ↕ ai-pricing-bus-update / case-library-bus-update
📡 AI定价基准库 ↔ 🗂️ 案例向量库
    ↓ simulator-bus-update
🎮 定价模拟器（5输入→4输出）
    ↓ risk-fuse-trigger
🚦 风控熔断（6阈值独立判定）
```

跨页同步机制：`localStorage写入 + storage事件监听`，7条事件通道：

| 事件名 | 触发源 | 消费方 |
|--------|--------|--------|
| funnel-bus-update | 信息漏斗 | 金融估值模拟器 |
| circle-bus-update | 画圈实验 | 创始人画像模块 |
| ai-pricing-bus-update | AI定价基准 | 定价模拟器 |
| case-library-bus-update | 案例向量库 | 估值系统+画像匹配 |
| pricing-bus-update | 反向拆解器 | CAPM向量覆盖 |
| simulator-bus-update | 定价模拟器 | 圆锥浓度修正 |
| risk-fuse-trigger | 风控熔断 | 全局告警 |

### 3.2 实现断层分析

#### 断层1：总线事件"声明了但未完全连通"

- info-funnel.html确实写入`window.funnelOutput`，但marketing-reinvented的消费端联动逻辑不一致
- 直接打开子页面时总线数据不存在，fallback逻辑各模块不统一——有的做了默认值填充，有的没有
- 跨页同步隐式依赖用户操作顺序（先开漏斗→再开估值器），无状态初始化保障

**实现完整度**：60%

#### 断层2：圆锥浓度模型——几何隐喻与数学实现的落差

文档宣称公式`C = m × ρ₀ / V(h)`（圆锥体积模型），但实际JS实现中：
- 浓度C是6个变量（毛利率、凯利、β、客户集中度、现金流、场景维度）的**线性加权和**
- 权重硬编码
- 0.48/0.50/0.68是判定阈值，定价乘数`Multiplier = 1 + (C - 0.48) × 2.5`是线性映射
- Canvas视觉上画了圆锥几何，但计算本质是加权评分

这不是bug，但这是**概念与实现的核心落差**——"圆锥"目前是视觉隐喻，不是真实几何计算。

**实现完整度**：70%

#### 断层3：案例库余弦相似度——数据完整但匹配半完成

- 16个案例的6维向量数据完整（瑞幸+7隐形冠军+8 AI）
- 余弦相似度+欧氏距离公式框架存在
- 但端到端用户流程不完整：缺少用户输入面板→实时计算→TopN排序→结果推送到估值器的完整闭环

**实现完整度**：50%

#### 断层4：画圈实验→风险向量——概念强连接，数据弱连接

circle-lab的三种模式（motor/model/market）、token流模拟、AB test框架输出的行为数据结构未明确定义：
- 画圈速度方差→风险偏好？
- 轨迹偏离度→执行纪律？
- 模式切换频率→认知灵活性？

README声称circle-bus-update→创始人画像，但这个映射目前是概念层面的，不是数据层面的。

**实现完整度**：30%

### 3.3 冗余分析

#### 冗余1：Canvas可视化重复造轮子

[general-game-os.html](../labs/evidence/general-game-os.html)中5个Canvas模块（博弈底座/低空经济/精算映射/物流底座/算力底座）：
- 结构完全同构（滑块→状态计算→Canvas绘制→数值展示）
- 但代码是5份独立实现，每份都有自己的网格绘制、动画循环、滑块绑定
- 后两个（物流/算力）README明确标注"结构完全同构"，代码却完全重复

#### 冗余2："三流分发"概念多处重复实现

信息分发/流量分发/物理分发框架在：
- README文档
- marketing-reinvented.html定价模拟器
- 瑞幸反向拆解器
- 多处讲解文案

概念统一，但代码实现各算各的，没有公共函数。

## 四、重构建议

### 4.1 第一优先级：技术债清偿（v1.5，1周）

**核心原则**：不换框架、不引入React/Vue、不加后端——保住"零依赖、双击即用"的最大优势。

#### 行动1：提取公共资源

```
/assets/
├── css/
│   ├── base.css          /* 重置、侧边栏、光晕、容器、动画 */
│   └── components.css    /* 卡片、按钮、滑块、模态框 */
└── js/
    ├── bus.js            /* localStorage数据总线（pub/sub统一封装） */
    ├── sidebar.js        /* 侧边栏JSON驱动渲染（不再复制HTML） */
    ├── glow.js           /* 鼠标跟随光晕 */
    ├── canvas-utils.js   /* 网格绘制、坐标轴、动画循环等公共Canvas工具 */
    ├── algorithms/
    │   ├── kelly.js           /* 凯利公式 */
    │   ├── capm.js            /* CAPM定价 */
    │   ├── risk-vector.js     /* 6维风险向量 */
    │   ├── concentration.js   /* 圆锥浓度模型 */
    │   └── similarity.js     /* 余弦相似度+欧氏距离 */
    └── data/
        ├── cases.js      /* 16案例6维向量 */
        ├── benchmarks.js /* AI定价4维基准 */
        └── nav-config.js /* 侧边栏导航配置JSON */
```

每个HTML只保留独有CSS/JS，公共部分用`<link>`和`<script src=>`引入。**预期代码量减少35-40%，改导航只改一个文件。**

#### 行动2：给算法层加JSDoc类型注释

不引入TS编译器，仅加类型标注：

```javascript
/**
 * 计算凯利最优比率
 * @param {number} p - 胜率 (0-1)
 * @param {number} b - 净赔率 (盈利/亏损)
 * @returns {number} f* 最优投入比例 (0-1)
 */
function kelly(p, b) {
  return Math.max(0, (b * p - (1 - p)) / b);
}
```

成本极低，收益极大——防止维度传错、向量算错。

#### 行动3：统一总线初始化

在bus.js中实现`initState()`：
1. 页面加载时检查localStorage各通道数据存在性
2. 不存在则填充合理默认值
3. 触发一次`bus-ready`事件
4. 所有模块等待该事件后再初始化，消除隐式操作顺序依赖

### 4.2 第二优先级：数据流断层补完（v1.6，1周）

#### 行动4：打通画圈实验→风险向量映射

定义circle-lab输出为标准化行为向量：
- 画圈速度方差 → 风险偏好维度
- 轨迹平均偏离度 → 执行纪律维度
- 三模式切换频率 → 认知灵活性维度
- 闭合率偏差 → 认知偏差维度

该向量直接作为6维风险向量的输入源，而非仅作为"创始人画像"隐喻。

#### 行动5：圆锥浓度模型真实几何化

将线性加权改为真实圆锥几何模型：
- 底面半径 r = f(市场规模、客户基数)
- 圆锥高度 h = f(壁垒厚度、技术浓度)
- 母线斜率 = f(定价权、毛利率)
- 体积 V = (1/3)πr²h
- 浓度 C = 实际体积 / 理论最大体积

0.48/0.50/0.68对应不同r/h比值下的体积比阈值，让"圆锥"成为有几何意义的计算模型而非纯视觉。

#### 行动6：案例库端到端匹配闭环

在case-library.html增加：
- 6个滑块（对应6维向量）的用户输入面板
- 实时计算与16案例的相似度
- Top3匹配案例高亮展示
- 通过case-library-bus-update将匹配结果推送到估值器，自动填充对标参数

### 4.4 绝对不要做的事（v2阶段禁令）

| 禁止项 | 原因 |
|--------|------|
| ❌ 引入React/Vue/任何前端框架 | 零依赖优势消失，退化为需要npm install的普通项目 |
| ❌ 加后端（Node/Python） | localStorage完全够用，加后端意味着部署成本、服务器成本、状态同步问题 |
| ❌ 做成SaaS多用户 | 项目最大力量来自"个人认知外骨骼"，多用户会稀释锐利度 |
| ❌ 为"代码优雅"牺牲算法透明度 | 宁可重复但逻辑清晰，不过度抽象导致公式被埋在设计模式里 |
| ❌ 大量补文档 | README已足够好，问题不是文档少而是不同步，建立"改代码顺手改README"习惯即可 |

### 5.2 核心诊断

这是一个**思想内核极其锐利、工程实现是原型级但能用、架构承载力已到极限**的项目。

它最珍贵的不是代码，是那套**圆锥浓度三条铁律**——0.48/0.50/0.68这三个阈值不是拍脑袋的，是从16个案例（瑞幸反例+7隐形冠军+8 AI公司）中提炼出来的判断标准，是有预测力的。

它目前最大的风险不是"代码写得乱"（原型阶段代码乱正常），而是**概念隐喻和可计算模型之间的落差正在扩大**：
- "圆锥"是隐喻不是几何
- "画圈→画像"是概念不是数据
- "7通道总线"是宣言不是全部连通

如果落差继续扩大，项目会退化成"视觉很酷的思想展示PPT"；如果收敛补完，它会成为一个真正能辅助商业决策的精算工具。

它最不需要的是"一个懂React的前端来重构代码"——那会杀死它的灵魂。它最需要的是：
1. 用最小工程手段止住代码腐烂（提取公共文件、统一总线）
2. 把概念隐喻一个个变成可计算的数学模型
3. 用一个端到端真实场景证明整个链路闭环

**方向是对的，内核是硬的，现在需要的是收敛，不是扩张。**
